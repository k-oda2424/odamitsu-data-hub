# 請求実績Excelインポート テスト計画書

## 対象機能
- 機能名: 請求実績Excelインポート
- 仕様書: `claudedocs/spec-invoice-import.md`
- API: `POST /api/v1/finance/invoices/import`
- バックエンド: `InvoiceImportService.importFromExcel()`
- フロントエンド: 請求書一覧画面 (`/finance/invoices`) のインポートDialog

## 1. バックエンド単体テスト (InvoiceImportService)

### 1.1 締日導出 (Row2パース)

| ID | テスト名 | 前提条件 | 手順 | 期待結果 |
|----|---------|---------|------|---------|
| BE-01 | Row2から末日締の締日を正しく導出する | Row2: `"2025年11月30日締 今回請求分"` | parseClosingDate()を呼び出す | `"2025/11/末"` が返る（11月末日=30、30==30なので「末」） |
| BE-02 | Row2から非末日締の締日を正しく導出する | Row2: `"2025年 7月20日締 今回請求分"` | parseClosingDate()を呼び出す | `"2025/07/20"` が返る（7月末日=31、20!=31なので日付） |
| BE-03 | 2月末日（28日）の締日を正しく導出する | Row2: `"2025年 2月28日締 今回請求分"` | parseClosingDate()を呼び出す | `"2025/02/末"` が返る（2025年2月末日=28） |
| BE-04 | 閏年2月末日（29日）の締日を正しく導出する | Row2: `"2024年 2月29日締 今回請求分"` | parseClosingDate()を呼び出す | `"2024/02/末"` が返る（2024年は閏年、2月末日=29） |
| BE-05 | 12月末日の締日を正しく導出する | Row2: `"2025年12月31日締 今回請求分"` | parseClosingDate()を呼び出す | `"2025/12/末"` が返る |

## 3. テスト優先度

| 優先度 | テストID | 理由 |
|--------|---------|------|
| 高 | BE-01, BE-02 | 締日導出はシステムの根幹ロジック |
| 高 | BE-08, BE-11 | 得意先コード変換は一意キーに直結 |
| 高 | BE-19, BE-21 | 松山999999スキップ / 第1事業部999999取込の分岐が業務要件の核心 |
| 高 | BE-23, BE-24, BE-25 | UPSERTとpayment_date保持は既存データ破壊防止に必須 |
| 高 | BE-30 | all-or-nothingトランザクションはデータ整合性の保証 |
| 高 | BE-26, BE-27, BE-28 | 不正ファイルの早期弾きで本番障害を防止 |
| 中 | BE-03, BE-04, BE-05 | 月末判定の境界値テスト |
| 中 | BE-06, BE-07 | NFKC正規化（SMILE出力形式が変わる可能性への備え） |
| 中 | BE-13, BE-14, BE-15 | shop_no判定（シンプルなロジックだが業務影響大） |
| 中 | FE-01〜FE-05 | フロントエンドの基本動作確認 |
| 低 | BE-09, BE-10, BE-12 | コード変換のバリエーション |
| 低 | BE-16, BE-17, BE-18, BE-20 | スキップ条件の網羅（基本的なnullチェック） |

## 5. 前提・制約

- バックエンド単体テストでは実際のExcelファイルを `src/test/resources/` に配置して読み込む
- UPSERTテスト（BE-23〜BE-25）はリポジトリのモックまたはテスト用DBで実施
- トランザクションテスト（BE-30）は `@SpringBootTest` で実DBを使用し、ロールバックを検証
- E2Eテストではファイルアップロードに `page.setInputFiles()` を使用
- E2EのインポートAPIモックは `page.route()` で `multipart/form-data` のPOSTを捕捉
