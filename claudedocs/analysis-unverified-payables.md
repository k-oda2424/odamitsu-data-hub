## 原因推定と推奨アクション

### 全体像

- **カテゴリ1 (1 件)**: DB にレコードが出来ているのに verification_result が NULL。
  - 現状は Cat1=1 だけ（ナカパック産業）。→ 再集計 or 手動マーク漏れ。
- **カテゴリ2 (9 件)**: そもそも t_accounts_payable_summary にレコードが無い。
  - 2A (2): 仕入ゼロ型。締め期間 2026-01-21〜2026-02-20 に購入が無い。例: 008100 大高製紙（2026-01-05 と 2026-02-26 に仕入有、期間内はゼロ）。
  - 2B (0): Excel code 有りで DB 無し（ロジック不備）。今期は 0 件。
  - 2D (7): Excel の supplier_code 空欄の行（運賃/事務費/個別仕入などの非 PAYABLE 項目）。これは正常挙動（DIRECT_PURCHASE に降格される設計）。

**⚠️ 真の構造的問題は『Cat2 の件数』ではなく『親NULL問題による金額漏れ』**。
Excel との件数差は少ないが、同じ支払先の中でも親行 NULL 分の仕入が抜けるため、支払金額と集計金額に慢性的な差額が出る。上記『パターンB 親NULL問題』セクション参照。

### 推奨アクション

1. **Cat1 対策**: `accountsPayableVerify` バッチ（または該当 verify エンドポイント）を 2026-02-20 で再実行。verify ボタンの実行漏れがないか UI でも確認。
2. **Cat2A 対策**: 
   - 仕入ゼロなのに支払がある仕入先は、集計ロジックを「仕入起点」から「(仕入 ∪ 直近支払予定)」に拡張する案 or 手動追加運用のいずれか。
   - 短期的にはオペレータが UI から手入力（verified_manually=true, verification_note='仕入ゼロ・前期繰延'）で登録。
3. **Cat2B 対策**: `TAccountsPayableSummaryService` の集計 SQL を点検し、以下を確認。
   - m_supplier.payment_supplier_no が NULL の行を除外していないか
   - 親行 payment_supplier_no が NULL で子行だけ保持している仕入先について、集計 key が親 code にフォールバックしていないか
   - 具体的には下記 m_supplier NULL 行一覧を参照。
4. **Cat2D 対策**: m_payment_supplier マスタに未登録コードを追加。

## 【重要】パターンB 親NULL問題: 仕入実績のある m_supplier.payment_supplier_no=NULL 行

`AccountsPayableSummaryCalculator.calculatePayableSummaries` (line 53-62)
は `m_supplier.payment_supplier_no IS NULL` な行の仕入明細を明示的に除外している。
この結果、親マスタに親行が残っているが子行にしか payment_supplier_no が設定されていない
supplier では、親行経由の仕入分が **買掛金集計から抜け落ちる**。

### 根本原因（コード参照）

`backend/src/main/java/jp/co/oda32/batch/finance/service/AccountsPayableSummaryCalculator.java` L53-62:

```java
// 親支払先(m_supplier.payment_supplier_no)未登録の仕入先は集計対象外（買掛金の突合キーが作れないため）
.filter(detail -> {
    MSupplier supplier = detail.getTPurchase().getMSupplier();
    if (supplier == null || supplier.getPaymentSupplierNo() == null) {
        ... missingPaymentSupplier.add(key); return false;  // ←集計から除外
    }
    return true;
})
```

### 修正案

1. **DB マスタ補完（推奨）**: m_supplier の親行 (`supplier_code=0XXX00`) に対して、同コードの子行と同じ `payment_supplier_no` を埋める。下記 114 件の多くは歴史的な親マスタ残骸なので UPDATE 一発で解消可能。
2. **Calculator 側フォールバック**: `supplier.getPaymentSupplierNo() == null` の場合、同一 `(shop_no, supplier_code)` プレフィクス（例えば 008100→008101/008105）の子を探索して payment_supplier_no を借りる or `m_payment_supplier` を `payment_supplier_code=supplier_code` で直接引く。
3. **監視**: 集計バッチログ `親支払先(payment_supplier_no)未設定のため集計対象外とした仕入先` を Slack 通知 or ダッシュボードに出力し、運用で気付けるようにする。
