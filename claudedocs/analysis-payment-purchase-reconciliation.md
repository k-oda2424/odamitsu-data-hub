## 7. 根本原因の仮説

### 仮説1: ユーザー指定の tx_month マッピングは実データと一致しない

- 依頼文では `振込み明細08-1-20.xlsx` (送金日 2026/1/20) → tx_month `2026-01-20` と指定されていたが、
  Excel の請求額と DB の `transaction_month=2025-12-20` が per-supplier でほぼ完全一致（79件 MATCH / 0 DIFF）した。
- つまり「送金日 M/5 も M/20 も同じ 前月20日締め 期間の買掛金を支払っている」。
  5日払いが早払い対象、20日払いがそれ以外、という業務分担になっている模様。
- 本レポートの数値は **データ整合性最優先の tx_month マッピング** で出している:
  - 2026/1/5  + 2026/1/20 払い → tx_month **2025-12-20**
  - 2026/2/5  + 2026/2/20 払い → tx_month **2026-01-20**

### 仮説2: `AccountsPayableSummaryCalculator` の消費税計算が「税率グループ合算×税率→HALF_DOWN」である

- 実装: `AccountsPayableSummaryCalculator.java#L92-100` — supplier×税率で subtotal を合計したあと、一度だけ税率を掛けて HALF_DOWN で整数化。
- 一方 SMILE 側 t_purchase_detail の `include_tax_subtotal` は明細ごとの四捨五入済み税込額の合計。
- よって税率別集計で下記の乖離が発生（単位: 円）:

| tx_month | 税率 | DB−(subtotal×(1+税率) HALF_DOWN, 同ロジックを当方で再計算) | この差は計算順序由来ではなく他要因の可能性 |
|----------|-----:|--------:|-------|
| 2025-12-20 | 8% | -29,788 | ※0 に近ければ計算ロジック一致 |
| 2025-12-20 | 10% | -203,850 | ※0 に近ければ計算ロジック一致 |
| 2026-01-20 | 8% | +0 | ※0 に近ければ計算ロジック一致 |
| 2026-01-20 | 10% | +2,747,544 | ※0 に近ければ計算ロジック一致 |
| 2026-02-20 | 8% | +22,680 | ※0 に近ければ計算ロジック一致 |
| 2026-02-20 | 10% | +3,224,097 | ※0 に近ければ計算ロジック一致 |

### 仮説3: `t_purchase_detail.include_tax_subtotal` に NULL 行があるが Calculator は `subtotal` のみ参照する

- 実装: `AccountsPayableSummaryCalculator.java#L73` で `subtotal` のみ集計。`include_tax_subtotal` は一切参照していない。→ NULL の有無は買掛集計の金額には影響しない。
- ただし `/purchases` 画面や本レポートの「仕入明細」列は `include_tax_subtotal` を SUM しているため、NULL があると買掛集計 vs 仕入明細の見かけ上の差が広がる。

| tx_month | include_tax_subtotal NULL 行数 |
|----------|----:|
| 2025-12-20 | 885 |
| 2026-01-20 | 351 |
| 2026-02-20 | 541 |

### 仮説4: Excel には仕入以外の経費（運送費・印刷・車両）が同じ表に混在している

- 名寄せ失敗14件の多くは `福山通運` `サクマ運輸㈱` `ティックトランスポート㈲` `ヨハネ印刷㈱` `リコージャパン㈱` `広島トヨペット㈱` `㈱ナカガワ`。
- これらは `m_supplier` / `m_payment_supplier` に未登録か「仕入先」として登録されていないため、買掛集計 DB（`t_purchase_detail` 集計）には出てこない。
- 月別合計差の大部分はこの **仕入でない経費行** で説明できる:
  - 2025-12-20: Excel−DB = +1,173,854 円 → 未マッチ合計 1,199,098 円（同月）で説明可。
  - 2026-01-20: Excel−DB = +350,391 円 → 未マッチ合計 411,642 円（同月）で説明可。

### 仮説5: 買掛金一覧の手動確定行（`verified_manually=true`）により Excel と DB の突合が過去実績で固定されている

- `verified_manually=true` 件数: 0 / 備考付き行: 0
- 手動確定行の Excel−DB 差は **再集計バッチが上書きしない**ため、`tax_included_amount_change` が固定化されている（`SmilePaymentVerifier` が手動行をスキップする設計 → MEMORY.md）。
- これは意図通りだが、元の `振込明細` 側の金額変更が DB に反映されず乖離として残る場合がある。

### 仮説6: `supplier_no=303` 除外ロジックの非対称性

- `AccountsPayableSummaryCalculator.java#L51` は `supplier.getPaymentSupplierNo() != 303` を条件にしている。
- つまり `m_supplier.payment_supplier_no` が 303 の仕入先の明細は除外される。`t_purchase.supplier_no=303` ではなく **payment_supplier_no=303** を除外している点に注意。
- `/purchases` 画面側の `PurchaseController.EXCLUDED_SUPPLIER_NO` は直接 `supplier_no` で除外しているので、**2 つの除外軸が異なる**。

### 仮説7: Excel 送り先名 ↔ 支払先マスタ の名寄せ失敗（仮説4 以外の純粋な表記揺れ）

- 名寄せ失敗件数: 14
- `コード`列が空 or 非数字で、かつ送り先名の部分一致もヒットしない行は突合不能。5-b 参照。

## 8. 推奨アクション

1. **計算順序の影響定量化**: `AccountsPayableSummaryCalculator` を「明細単位四捨五入→合計」版と「合計→税率→HALF_DOWN」版で並行実行し、全支払先で差額を出力する検証バッチを一度だけ流す（tx_month ごとに数十円〜数百円単位の差が閾値 100 円内に収まるか要確認）。
2. **`include_tax_subtotal` NULL の解消**: SMILE 取込側で `t_purchase_detail.include_tax_subtotal` を必ず埋める、または本カラムへの依存を排除（Calculator は既に参照していないので、`/purchases` 画面側の集計も `subtotal + round(subtotal*tax)` に揃える）。
3. **名寄せロバスト化**: `PaymentMfImportService` のコードマッチを `m_supplier.supplier_code` も検索対象にする / `m_payment_supplier` のエイリアス表（`m_payment_mf_rule` の payee 同義語）を突合 API に流用する。
4. **除外軸の統一**: `FinanceConstants.EXCLUDED_SUPPLIER_NO` を「payment_supplier_no で除外」「supplier_no で除外」のどちらかに統一し、ドキュメント明記（現状は Calculator と PurchaseController で意味が異なる）。
5. **突合レポート UI 化**: 本スクリプトをバックエンドバッチ（`PaymentReconciliationTasklet` 的）として実装し、`/finance/accounts-payable` 画面から差異一覧をエクスポートできるようにする。
