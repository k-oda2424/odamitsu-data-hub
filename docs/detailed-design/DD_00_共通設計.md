## 1. 共通基盤設計

### 1.1 アプリケーション起動方式

本システムは1つの実行可能 JAR（`app.jar`）に Web アプリケーションとバッチアプリケーションを同梱し、Spring Profile によって起動モードを切り替える設計である。

#### 1.1.1 ビルド成果物

`build.gradle` の `bootJar` タスクにより、`app.jar` が生成される。メインクラスは `jp.co.oda32.WebApplication` に固定されている。

```groovy
bootJar {
    launchScript()
    archiveFileName = 'app.jar'
    mainClassName = 'jp.co.oda32.WebApplication'
}
```

#### 1.1.2 Web アプリケーションエントリポイント

| 項目 | 内容 |
|---|---|
| クラス名 | `jp.co.oda32.WebApplication` |
| ファイル | `src/main/java/jp/co/oda32/WebApplication.java` |
| アノテーション | `@SpringBootApplication`, `@Profile("web")` |
| 起動コマンド | `java -jar app.jar --spring.profiles.active=web,dev` |

**処理フロー:**

1. `main(String[] args)` が呼び出される
2. `SpringApplication.run(WebApplication.class, args)` を実行
3. Spring Boot が `web` プロファイルの設定ファイル群を読み込み
4. 組み込み Tomcat サーバーを起動し、Servlet コンテナとして動作
5. `SecurityConfig`, `WebConfig`, `ApiConfig` 等の `@Configuration` クラスが順次初期化される

```java
@SpringBootApplication
@Profile("web")
public class WebApplication {
    public static void main(String[] args) {
        SpringApplication.run(WebApplication.class, args);
    }
}
```

#### 1.1.3 バッチアプリケーションエントリポイント

| 項目 | 内容 |
|---|---|
| クラス名 | `jp.co.oda32.BatchApplication` |
| ファイル | `src/main/java/jp/co/oda32/BatchApplication.java` |
| アノテーション | `@SpringBootApplication`, `@Profile("batch")`, `@EnableTransactionManagement` |
| 起動コマンド | `java -jar app.jar --spring.profiles.active=batch,dev --spring.batch.job.name=<ジョブ名>` |
| Web コンテキスト | `WebApplicationType.NONE`（無効化） |

**処理フロー:**

```
1. main(String[] args) が呼び出される
2. コマンドライン引数を解析:
   a. --spring.batch.job.name=<ジョブ名> からジョブ名を取得
   b. shopNo=<値>, warehouseNo=<値> 等の追加パラメータを取得
   c. 第一引数がオプションプレフィックスなしの場合、ジョブ名として扱う
3. ジョブ名が未指定の場合 → エラーメッセージ出力し System.exit(1)
4. SpringApplication インスタンスを生成
5. WebApplicationType.NONE を設定（Web コンテキスト無効化）
6. app.run(args) でコンテキスト起動
7. JobLauncher Bean をコンテキストから取得
8. Bean 名 = ジョブ名 + "Job" で Job Bean を検索
   - 例: ジョブ名 "purchaseFileImport" → Bean 名 "purchaseFileImportJob"
9. Bean が見つからない場合 → エラー出力し System.exit(1)
10. JobParametersBuilder でパラメータを構築:
    a. time=System.currentTimeMillis() を自動付与（重複実行防止）
    b. "--" プレフィックス付き引数（spring.batch.* と application.* を除く）を追加
    c. "key=value" 形式の引数も追加
11. jobLauncher.run(job, parameters) でジョブ実行
12. finally ブロックで SpringApplication.exit(context) → System.exit() を呼出
```

**除外クラス設定:**

`@ComponentScan` の `excludeFilters` で旧バッチエントリポイントを除外している。

```java
@ComponentScan(basePackages = {"jp.co.oda32"},
    excludeFilters = {@ComponentScan.Filter(
        type = FilterType.ASSIGNABLE_TYPE,
        classes = {
            AccountsPayableSummaryBatch.class,
            AccountsReceivableSummaryBatch.class,
            PurchaseJournalIntegrationBatch.class
        }
    )})
```

### 1.3 ApplicationType 条件付き Bean 登録

`@ApplicationType` アノテーションと `ApplicationTypeCondition` クラスにより、`application.type` プロパティの値に基づいて Bean の登録可否を制御する。

#### 1.3.1 @ApplicationType アノテーション

| 項目 | 内容 |
|---|---|
| ファイル | `src/main/java/jp/co/oda32/annotation/ApplicationType.java` |
| 適用対象 | `ElementType.TYPE`, `ElementType.METHOD` |
| 保持ポリシー | `RetentionPolicy.RUNTIME` |
| 条件クラス | `ApplicationTypeCondition` |

```java
@Target({ElementType.TYPE, ElementType.METHOD})
@Retention(RetentionPolicy.RUNTIME)
@Conditional(ApplicationTypeCondition.class)
public @interface ApplicationType {
    String[] value();
}
```

#### 1.3.2 ApplicationTypeCondition 実装

| 項目 | 内容 |
|---|---|
| ファイル | `src/main/java/jp/co/oda32/annotation/ApplicationTypeCondition.java` |
| インターフェース | `org.springframework.context.annotation.Condition` |

**判定ロジック:**

1. `@ApplicationType` アノテーションの `value` 属性（文字列配列）を取得
2. `Environment` から `application.type` プロパティの値を取得
3. アノテーションの value 配列にプロパティ値が含まれていれば `true`（Bean 登録）
4. 含まれていなければ `false`（Bean 登録しない）

**使用例:** `ShopCheckAop` クラスは `@ApplicationType("web")` が付与されており、`application.type=web` の場合のみ Bean として登録される。

#### getShopNo() メソッド実装

```java
@Override
public Integer getShopNo() {
    MCompany mCompany = getCompany();
    if (mCompany == null) {
        return null;
    }
    MPartner partner = mCompany.getPartner();
    if (partner != null) {
        // 得意先の在庫の場合、所属 shopNo を返す
        return partner.getShopNo();
    }
    // ショップの在庫の場合
    return mCompany.getShopNo();
}
```

**処理フロー:**

```
1. getCompany() で MCompany を取得
2. MCompany が null の場合 → null を返す
3. MCompany から MPartner を取得
4. MPartner が null でない場合（得意先に紐づくエンティティ）
   → partner.getShopNo() を返す（得意先の担当店舗番号）
5. MPartner が null の場合（ショップ直属のエンティティ）
   → mCompany.getShopNo() を返す
```

**設計意図:** 得意先（partner）に紐づくエンティティの場合、その得意先を担当する店舗番号を返すことで、担当ショップユーザーのみがデータにアクセスできるようにする。

3. entity.setAddDateTime(DateTimeUtil.getNow()) で現在日時を設定
4. del_flg が null の場合、Flag.NO（"0"）を設定

#### validateUpdateByShop メソッドの処理フロー

```
1. entity.getShopNo() を取得
2. entityShopNo が null の場合 → チェックなし（return）
3. loginUser の shopNo が null の場合 → 例外 throw
4. loginUser の shopNo が 0（管理者）の場合 → チェックなし（return）
5. entityShopNo と loginUser の shopNo が不一致の場合 → 例外 throw
```

### 3.2 validateGet メソッド（単一エンティティ取得時のチェック）

| 項目 | 内容 |
|---|---|
| アドバイスタイプ | `@Around` |
| ポイントカット | `execution(* *..*Service.*get*(..))` |
| 対象 | Service クラスの `get` を含む全メソッド |
| 戻り値 | `Object`（元メソッドの戻り値またはフィルタリング結果） |

**処理フロー:**

```
1. pj.proceed() で対象メソッドを実行し、戻り値を取得
2. 戻り値が IEntity ではない場合 → そのまま返す（非エンティティはチェック対象外）
3. IEntity にキャストし、entity.getShopNo() を取得
4. getShopNo() が null の場合 → そのまま返す（店舗に紐づかないエンティティ）
5. LoginUserUtil.getLoginUserInfo() でログインユーザーの shopNo を取得
6. loginUserShopNo が null の場合 → null を返す（取引先ユーザー等のアクセス拒否）
7. loginUserShopNo == 0 の場合 → そのまま返す（管理者は全データアクセス可能）
8. entity.getShopNo() == loginUserShopNo の場合 → そのまま返す（自店舗データ）
9. 上記以外 → null を返す（他店舗データへのアクセス拒否）
```

```
[Client] → [Service.get*()] → [ShopCheckAop.validateGet()]
                                      │
                                      ├─ IEntity でない → そのまま返す
                                      ├─ shopNo null → そのまま返す
                                      ├─ loginUserShopNo null → null（拒否）
                                      ├─ loginUserShopNo == 0 → そのまま返す（管理者）
                                      ├─ shopNo 一致 → そのまま返す
                                      └─ shopNo 不一致 → null（拒否）
```

### 3.3 validateFind メソッド（リスト取得時のチェック）

| 項目 | 内容 |
|---|---|
| アドバイスタイプ | `@Around` |
| ポイントカット | `execution(* *..*Service.*find*(..))` |
| 対象 | Service クラスの `find` を含む全メソッド |
| 戻り値 | `List`（フィルタリング後のリスト） |

**処理フロー:**

```
1. pj.proceed() で対象メソッドを実行
   - ClassCastException 等が発生した場合 → ログ出力し null を返す
2. 戻り値が null の場合 → null を返す
3. リスト内の全要素が IEntity かチェック
   - IEntity でない要素がある場合 → そのまま返す（非エンティティリスト）
4. リスト内で getShopNo() が null の要素がある場合 → そのまま返す
5. LoginUserUtil.getLoginUserInfo() でログインユーザーの shopNo を取得
6. loginUserShopNo が null または -1 の場合 → null を返す（アクセス拒否）
7. loginUserShopNo == 0 の場合 → そのまま返す（管理者）
8. Stream でフィルタリング:
   - entity.getShopNo() != null かつ entity.getShopNo() == loginUserShopNo の要素のみ残す
9. フィルタリング後のリストを返す
```

### 3.4 アクセス制御マトリクス

| ユーザー種別 | shopNo | get* の振る舞い | find* の振る舞い |
|---|---|---|---|
| 管理者（ADMIN） | 0 | 全エンティティ返却 | 全エンティティ返却 |
| ショップ（SHOP） | 1-3 | 自店舗のみ返却、他店舗は null | 自店舗のみフィルタリング |
| 取引先（PARTNER） | 所属店舗の shopNo | 自店舗のみ返却 | 自店舗のみフィルタリング |
| shopNo null ユーザー | null | null 返却（拒否） | null 返却（拒否） |

| セッション生成ポリシー | `IF_REQUIRED` | `SecurityConfig` |

#### 6.4.2 セッションタイムアウト検知フロー

**関連クラス:**

| クラス | 役割 |
|---|---|
| `SessionTimeoutFilter` | セッションタイムアウト前のアクセスURL記録 |
| `SessionTimeoutRedirectStrategy` | セッション無効時のリダイレクト処理 |
| `CustomAuthenticationSuccessHandler` | 再ログイン後の復元リダイレクト |

**処理シーケンス:**

```
[ユーザーリクエスト]
    │
    ▼
[SessionTimeoutFilter] (UsernamePasswordAuthenticationFilter の前に挿入)
    │
    ├─ 認証済み & GET & 非除外URL & 非Ajax の場合:
    │   └─ session.setAttribute("LAST_ACCESSED_URL", 現在のURL+クエリ)
    │
    ├─ 未認証の場合:
    │   └─ session から "LAST_ACCESSED_URL" を取得し
    │      "LAST_ACCESSED_URL_BEFORE_TIMEOUT" に移動
    │
    ▼
[filterChain.doFilter()]
    │
    ▼
[Spring Security チェーン]
    │
    ├─ セッション無効検知時:
    │   └─ [SessionTimeoutRedirectStrategy.onInvalidSessionDetected()]
    │       ├─ Ajax リクエスト → 401 + JSON {"error":"Session timeout"} を返す
    │       ├─ 既にログイン画面へのアクセス → forward（リダイレクトループ防止）
    │       └─ 通常リクエスト → /loginForm?timeout=true にリダイレクト
    │
    └─ 再ログイン成功時:
        └─ [CustomAuthenticationSuccessHandler.onAuthenticationSuccess()]
            ├─ "LAST_ACCESSED_URL_BEFORE_TIMEOUT" がある場合
            │   └─ その URL にリダイレクト（ログイン関連URL除外）
            ├─ Spring Security の SavedRequest がある場合
            │   └─ その URL にリダイレクト
            └─ それ以外 → /dashboard にリダイレクト
```

**除外 URL パターン（SessionTimeoutFilter）:**

```
/loginForm, /login, /logout, /error, /dashboard,
/css/, /js/, /images/, /fonts/, /favicon.ico, /api/, /batch
```

**ログイン関連 URL 判定（CustomAuthenticationSuccessHandler）:**

以下を含む URL はリダイレクト先として使用しない:
- `/loginForm`, `/login`, `/logout`, `/error`, `/`（ルート）

## 7. トランザクション管理方針

### 7.1 JPA/Hibernate のトランザクション境界

#### 7.1.1 Web アプリケーション

| 設定項目 | 値 | 説明 |
|---|---|---|
| `spring.jpa.open-in-view` | `true`（本番） | Open Session in View パターン。ビューレンダリング中も Hibernate セッションを保持 |
| トランザクション制御 | Spring `@Transactional` | Service 層メソッド単位でトランザクション境界を設定 |

**標準的なトランザクション境界:**

```
[Controller]
    │
    ▼
[Service メソッド] ← @Transactional でトランザクション開始
    │
    ├─ Repository.save() / findAll() 等
    │
    └─ メソッド終了 → トランザクション commit
        ※ 例外発生時 → rollback
```

#### 7.1.2 バッチアプリケーション

| 設定項目 | 値 | 説明 |
|---|---|---|
| `@EnableTransactionManagement` | 有効 | BatchApplication に明示的に設定 |
| `spring.datasource.hikari.auto-commit` | `false` | HikariCP の自動コミットを無効化 |

### 7.2 DDL（TRUNCATE）の扱い

TRUNCATE はトランザクション内でロールバック不可な DDL 操作である。JPA のトランザクション管理とは別に処理する必要がある。

**実装方針:**

```
1. JdbcTemplate で直接 TRUNCATE SQL を実行
2. JPA トランザクションの外で処理する
3. Tasklet の冒頭で TRUNCATE を実行し、その後の DML はトランザクション管理下に置く
```

**対象テーブル例:**

| テーブル名 | TRUNCATE タイミング |
|---|---|
| `w_smile_purchase_output_file` | 仕入ファイル取込ジョブの先頭 Step |
| `w_smile_payment` | SMILE 支払情報取込の先頭 Step |
| `t_accounts_payable_summary` | 買掛金集計ジョブの初期化 Step |

### 7.3 バッチ処理のトランザクション設計

#### 7.3.1 Chunk 処理のトランザクション

```
[Step 実行]
    │
    ▼
[Chunk トランザクション] ← チャンクサイズ単位でトランザクション開始
    │
    ├─ Reader: N 件読み込み
    ├─ Processor: 加工処理
    └─ Writer: 書き込み → commit
        │
        └─ 例外発生時 → rollback（チャンク単位）
```

| ジョブ | チャンクサイズ | 理由 |
|---|---|---|
| 仕入ファイル取込 | 500 | 大量データの効率的処理 |
| SMILE 受注取込 | 500 | 大量データの効率的処理 |
| 商品ファイル取込 | 10 | 商品データの慎重な処理 |
| SMILE 支払取込 | 100 | 中量データの処理 |

#### 7.3.2 Tasklet 処理のトランザクション

```
[Step 実行]
    │
    ▼
[Tasklet トランザクション] ← Step 単位でトランザクション
    │
    ├─ execute() メソッド実行
    └─ RepeatStatus.FINISHED → commit
        │
        └─ 例外発生時 → rollback
```

#### 7.3.3 ジョブパラメータによる重複実行防止

Spring Batch は同一ジョブパラメータでの重複実行を許可しない仕様である。`BatchApplication` では `time=System.currentTimeMillis()` を自動付与することで毎回ユニークなパラメータを生成する。

```java
JobParametersBuilder parametersBuilder = new JobParametersBuilder();
parametersBuilder.addLong("time", System.currentTimeMillis());
```

### 7.4 トランザクション管理の注意事項

| 項目 | 方針 |
|---|---|
| StackOverflowError | カスタムトランザクション管理を避け、Spring Batch のデフォルト管理を使用 |
| TRUNCATE 実行 | JdbcTemplate で JPA トランザクション外に実行 |
| バッチ auto-commit | `false` に設定し、明示的コミットを実施 |
| Open-in-view | 本番環境で `true`（遅延ロード対応） |

## 9. エラーハンドリング方針

### 9.1 Web アプリケーションの例外処理パターン

### 9.2 バッチアプリケーションの例外処理パターン

#### 9.2.1 ジョブ実行時の例外処理

```
BatchApplication.main()
    │
    ├─ ジョブ名未指定 → エラーメッセージ出力、System.exit(1)
    ├─ Bean が見つからない → エラーメッセージ出力、System.exit(1)
    ├─ jobLauncher.run() で例外発生 → エラーメッセージとスタックトレースを出力
    └─ finally → SpringApplication.exit() + System.exit()
```

#### 9.2.2 Chunk 処理の例外処理

Spring Batch のチャンク処理では、チャンク単位でトランザクションが管理される。

| 例外発生箇所 | 処理 |
|---|---|
| Reader | チャンク全体がロールバック。Step が FAILED に遷移 |
| Processor | チャンク全体がロールバック。Step が FAILED に遷移 |
| Writer | チャンク全体がロールバック。Step が FAILED に遷移 |

スキップ設定がある場合（例: SMILE 支払取込のスキップ上限 10000 件）、スキップ可能な例外はスキップされ、上限を超えた場合に Step が失敗する。

#### 9.2.3 Tasklet 処理の例外処理

| 例外パターン | 処理 |
|---|---|
| Tasklet の `execute()` で例外スロー | Step がロールバックされ FAILED に遷移 |
| TRUNCATE 実行時の例外 | DDL のため部分ロールバック不可。Step が FAILED に遷移 |

#### 9.2.4 ExitStatusChangeListener による終了ステータス制御

`ExitStatusChangeListener` は `afterStep()` で常に `ExitStatus.COMPLETED` を返す。これにより、Step でエラーが発生しても後続の Step が実行される設計を実現する。ただし、この動作は全 Step に適用されるため、エラーハンドリングの意図を明確にする必要がある。

### 9.3 エラーハンドリング方針のまとめ

| 層 | 方針 |
|---|---|
| Controller | try-catch で Service 例外をハンドリングし、エラーメッセージを Model に設定してビューに返す |
| Service（CustomService） | 店舗権限チェック違反は `Exception` をスロー。バッチ実行時はログインユーザーが存在しないため `batchId` で代替 |
| AOP | アクセス拒否は `null` 返却で表現。ClassCastException はログ出力して `null` を返す |
| Filter | 例外はスルーし、リクエスト処理の継続を優先 |
| バッチ | Spring Batch のデフォルトトランザクション管理に委ね、カスタムトランザクション管理は避ける |
| バッチリスナー | ジョブ開始・終了のコンソールログ出力。ExitStatusChangeListener で Step 終了ステータスを COMPLETED に強制 |
