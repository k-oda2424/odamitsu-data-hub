```java
public BigDecimal getPurchaseCaseNum() {
    if (BigDecimalUtil.isZero(BigDecimalUtil.convertNullToZero(containNum))) {
        return null;
    }
    return goodsNum.divide(containNum, RoundingMode.DOWN);
}
```

- **入力**: `goodsNum`（数量）、`containNum`（入数）
- **出力**: ケース数（端数切捨て）。`containNum`が0またはnullの場合は`null`を返す
- **丸めモード**: `RoundingMode.DOWN`

#### 履歴記録メカニズム

`MPurchasePriceService.save()` 内部で `BeanUtils.copyProperties()` により `MPurchasePrice` から `MPurchasePriceLog` へフィールドをコピーし、`MPurchasePriceLogService.save()` でINSERTする。

```java
// MPurchasePriceService.save() の実装
public MPurchasePrice save(MPurchasePrice purchasePrice) throws Exception {
    MPurchasePrice mPurchasePrice = this.insert(this.mPurchasePriceRepository, purchasePrice);
    MPurchasePriceLog purchasePriceLog = new MPurchasePriceLog();
    BeanUtils.copyProperties(mPurchasePrice, purchasePriceLog);
    this.mPurchasePriceLogService.save(purchasePriceLog);
    return mPurchasePrice;
}
```

### 4.4 変更ログ記録ロジック

`MPurchasePriceService.save()` メソッド内で自動的に実行される。

```java
public MPurchasePrice save(MPurchasePrice purchasePrice) throws Exception {
    // 1. 仕入価格をINSERT/UPDATE
    MPurchasePrice mPurchasePrice = this.insert(this.mPurchasePriceRepository, purchasePrice);
    // 2. 履歴テーブル用エンティティを生成

    return mPurchasePrice;
}
```

### 6.2 ShopNoAwareItemReader の設計

**クラス**: `jp.co.oda32.batch.purchase.ShopNoAwareItemReader`
**継承**: `MultiResourceItemReader<PurchaseFile>`
**スコープ**: Bean定義時に設定

#### 取込対象ファイル

`m_shop_linked_file.smile_purchase_file_name` に登録された全ファイルを順次取込。
複数事業部に対応して CSV は複数本投入される。

| shop_no | ファイル例 | 備考 |
|---|---|---|
| 1 | `purchase_import.csv` | 第1事業部 SMILE（shori_renban 330000台） |
| 2 | `purchase_import2_YYYYMMDD.csv` | 第2事業部 SMILE（shori_renban 80000台） |

`(ext_purchase_no, shop_no)` が実質ユニークキー。shop_no が違えば shori_renban が衝突しても別伝票として扱う。

#### 動作原理（重要: リソース追跡の仕組み）

`MultiResourceItemReader` は現在読込中の `Resource` を公開 API として提供しないため、
`setDelegate(ResourceAwareItemReaderItemStream)` で設定される delegate を本クラス側でラップし、
`setResource()` の呼び出しを契機に現在リソースを追跡する。

```
1. コンストラクタで List<MShopLinkedFile> を受け取る
   - MShopLinkedFile には smilePurchaseFileName（ファイルパス）と shopNo が含まれる
2. setDelegate(delegate) を override:
   - delegate を ResourceTrackingDelegate でラップして super に渡す
   - ResourceTrackingDelegate.setResource(resource) が呼ばれると currentResource を更新
3. read() メソッドをオーバーライド:
   a. super.read() で1行読込（PurchaseFile オブジェクト取得）
   b. currentResource（= ファイル切替時に記録された直近リソース）から filename を取得
   c. shopLinkedFileList をストリームでフィルタ:
      - file.getSmilePurchaseFileName() からファイル名部分を抽出（Paths.get().getFileName()）
      - filename と一致するものを検索
   d. 一致した場合: purchaseFile.setShopNo(matched.getShopNo())
   e. 一致しない場合: WARN ログ出力（shop_no 未設定で返却）
4. ショップ番号が設定された PurchaseFile を返却
```

#### 過去のバグと修正（重要）

**旧実装（バグあり）**: `resources[]` 配列の先頭から一致検索していたため、複数ファイル取込時に
全行が先頭ファイルの `shop_no` になるバグがあった。shop_no=2 で登録すべき行が shop_no=1 に
複製され、買掛金集計が二重計上される原因となっていた。

**現在の実装**: `setResource()` フックで「今どのファイルを読んでいるか」を追跡し、1行ごとに
正しいファイル名 → shop_no を決定する。`ResourceTrackingDelegate` は `setResource`/`read`/
`open`/`update`/`close` をすべて内包 delegate に委譲し、`setResource` 呼び出し時のみ
`currentResource` を記録する副作用を持つ。

#### ファイル名マッチングロジック

```java
String filename = currentResource.getFilename();
MShopLinkedFile matched = shopLinkedFileList.stream()
        .filter(f -> filename.equals(
                Paths.get(f.getSmilePurchaseFileName()).getFileName().toString()))
        .findFirst()
        .orElse(null);
```

### 6.4 SmilePurchaseImportTasklet のUPSERTロジック

**ステップ3**: ワークテーブル `w_smile_purchase_output_file` から `t_purchase` / `t_purchase_detail` への UPSERT

```
1. ワークテーブルから新規/更新候補レコードを取得
   - findNewPurchases() / findModifiedPurchases()（ページング）
   - いずれも shouhin_code NOT IN ('00000021','00000023') を条件に含む
2. 処理連番（ext_purchase_no）+ ショップ番号でグルーピング
3. 各グループ（= 1仕入伝票）に対して:
   a. TPurchaseService.getByUniqKey(shopNo, processingSerialNumber) で既存チェック
   b. 既存なし → INSERT（新規仕入伝票）
   c. 既存あり → UPDATE（既存仕入伝票の更新）
   d. 仕入明細も同様にUPSERT
```

**重要: 第2事業部月次集約行の除外**

ワークテーブル自体は SMILE から来た CSV を verbatim で保持する方針のため、除外はこの
検索段階（`findNewPurchases` / `findModifiedPurchases`）で `AND wspof.shouhin_code NOT IN
('00000021','00000023')` により行う。

| goods_code | 意味 |
|---|---|
| `00000021` | 第2事業部 10%課税商品の月次集約（経理が第1事業部 SMILE に手入力） |
| `00000023` | 第2事業部 8%課税商品の月次集約（同上） |

設計思想: 買掛集計は shop_no=2 の個別仕入（生データ）を源泉とするため、月次集約の手入力行は
二重計上になる。本テーブルに入れない運用に統一し、集計フェーズも防御フィルタとして
同じ除外を行う（二重の安全策）。定数は `FinanceConstants.DIVISION2_AGGREGATE_GOODS_CODES`。

### 6.5 PurchaseLinkSendOrderTasklet の紐付けロジック

**ステップ4**: 仕入明細と発注明細の自動マッチング

#### 処理フロー

```
1. 発注明細を取得:
   - ステータスが「発注済(00)」「納期回答(10)」「入荷済(20)」のもの
   - tSendOrderDetailService.findBySendOrderDetailStatusList()
2. 仕入明細を取得:
   - 発注番号が未紐付け（sendOrderNo IS NULL）
   - 1ヶ月以内の仕入明細
   - tPurchaseDetailService.findByNotInputSendOrder(LocalDateTime.now().minusMonths(1).toLocalDate())
3. マッチングルール（5条件すべて一致）:
   条件1: purchaseDetail.sendOrderNo == null（未紐付け）
   条件2: purchaseDetail.goodsNo != null
   条件3: purchaseDetail.goodsNo == sendOrderDetail.goodsNo（同商品）
   条件4: purchaseDetail.goodsNum == sendOrderDetail.sendOrderNum（同数量）
   条件5: purchaseDetail.purchaseDate > sendOrder.sendOrderDateTime.toLocalDate()（仕入日 > 発注日）
4. マッチング時の処理:
   a. 仕入明細に sendOrderNo, sendOrderDetailNo をセット
   b. 発注明細ステータスが ARRIVED(20) の場合:
      - 仕入明細の stockProcessFlg = Flag.YES（在庫処理済）
   c. 発注明細の arrivedDate が null → 仕入明細の purchaseDate を補完
   d. 発注明細の arrivedNum が null → 仕入明細の goodsNum を補完
   e. 発注明細ステータスを PURCHASED(30) に更新
5. マッチングした明細を一括更新:
   - tSendOrderDetailService.update(updateSendOrderDetailList)
   - tPurchaseDetailService.update(updatePurchaseDetailList)
```

### 6.6 PurchasePriceCreateTasklet の処理

**ステップ5**: 仕入実績からの仕入単価マスタ自動更新

#### 処理フロー

```
1. tPurchaseDetailService.findLatestPurchasePrice() で未反映仕入明細を取得
   - JPQL条件:
     a. 同一商品コードで最新の仕入日のレコードのみ（NOT EXISTS サブクエリ）
     b. goodsNo IS NOT NULL
     c. purchasePriceReflect = '0' OR purchasePriceReflect IS NULL
     d. goodsNum > 0
     e. goodsPrice > 0
   - ORDER BY goodsCode
2. 空の場合: "仕入価格マスタ作成する仕入がありません。処理終了。" → FINISHED
3. フィルタリング:
   a. goodsPrice > 0（値引きデータ除外）
   b. goodsNum > 0（数量未入力除外）
4. convertTPurchaseDetailToMPurchasePrice() で変換:
   - supplierNo = tPurchaseDetail.getTPurchase().getSupplierNo()
   - goodsNo = tPurchaseDetail.getGoodsNo()
   - shopNo = tPurchaseDetail.getShopNo()
   - partnerNo = 0（直接仕入）
   - destinationNo = 0（デフォルト）
   - goodsPrice = tPurchaseDetail.getGoodsPrice()
   - taxCategory = tPurchaseDetail.getTaxCategory()
   - includeTaxFlg = Flag.NO.getValue()（税抜）
   - taxRate = tPurchaseDetail.getTaxRate()
   - note = tPurchaseDetail.getTPurchase().getNote()
   - lastPurchaseDate = tPurchaseDetail.getPurchaseDate()
5. distinct() で重複除去
6. mPurchasePriceService.insert(mPurchasePriceList) で一括UPSERT（履歴記録含む）
7. tPurchaseDetailService.updateAllPurchasePriceReflect() で反映フラグ一括更新
   - UPDATE文: purchasePriceReflect = '1' WHERE purchasePriceReflect = '0' OR purchasePriceReflect IS NULL
```
