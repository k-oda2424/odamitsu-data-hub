
#### 1.1.4 金額カラムの使い分け

| カラム | 用途 | 更新タイミング |
|--------|------|---------------|
| `taxIncludedAmountChange` | バッチ集計時の作業値。SMILE照合で調整される | 集計バッチ/検証バッチ |
| `taxIncludedAmount` | MF CSV出力後の確定値（SMILE支払額スナップショット） | CSV出力バッチ |
| `taxExcludedAmountChange` | 税抜作業値 | 集計バッチ |
| `taxExcludedAmount` | 税抜確定値 | CSV出力バッチ |
| `verifiedAmount` | 振込明細/手入力の税込請求額（検証比較元） | 手動検証・振込明細MF一括検証 |

**データフロー**:
```
集計バッチ → taxIncludedAmountChange に書込
  ↓
SMILE再検証バッチ → taxIncludedAmountChange を調整（verifiedManually=true の行はスキップ）
  ↓
振込明細MF一括検証 or 手動検証 → verifiedAmount / verificationNote を書込、verifiedManually=true
  ↓
CSV出力バッチ → taxIncludedAmountChange → taxIncludedAmount にコピー（確定）
```

`verifiedAmount` は `tax_included_amount`（MF出力用スナップショット）とは独立で、
手動・振込明細ベースの検証値を保持する。100円閾値での一致判定はこの値と
`taxIncludedAmountChange` を比較する。

## 2. 買掛金集計画面処理設計

### 2.2 一覧画面処理フロー

**エンドポイント**: `GET /finance/accountsPayable`
**メソッド**: `list(@RequestParam(required = false) String transactionMonth, Model model)`
**テンプレート**: `finance/accountsPayable/list`

#### 2.2.1 取引月の自動算出ロジック

```
入力: transactionMonth パラメータ（yyyy-MM形式、任意）
処理:
  IF transactionMonth が指定されている THEN
    date = LocalDate.parse(transactionMonth + "-01")
  ELSE
    date = LocalDate.now().withDayOfMonth(20)   // 当月20日
    IF LocalDate.now().getDayOfMonth() < 20 THEN
      date = date.minusMonths(1)                // 20日前なら前月20日
    END IF
  END IF
出力: date (LocalDate)
```

**例**:
- 現在日が2026-02-10 → date = 2026-01-20（前月20日）
- 現在日が2026-02-25 → date = 2026-02-20（当月20日）
- transactionMonth="2025-04" → date = 2025-04-01

#### 2.2.2 処理シーケンス

```
1. 取引月を算出（上記ロジック）
2. tAccountsPayableSummaryService.findByTransactionMonth(date) でサマリー全件取得
3. mSupplierService.findAll() で仕入先マスタをMap化
   └ key: supplierNo, value: MSupplier（重複時は先勝ち）
4. サマリーを2グループに分割:
   a. 検証済み: verificationResult != null のデータ
   b. 未検証: verificationResult == null のデータ
5. 不一致件数をカウント: verificationResult == 0 の件数
6. Modelに設定:
   - transactionMonth (yyyy-MM形式)
   - verifiedSummaries
   - unverifiedSummaries
   - supplierMap
   - unmatchedCount
```

### 2.3 詳細画面処理フロー

**エンドポイント**: `GET /finance/accountsPayable/{shopNo}/{supplierNo}/{transactionMonth}/{taxRate}`
**メソッド**: `detail(@PathVariable Integer shopNo, @PathVariable Integer supplierNo, @PathVariable String transactionMonth, @PathVariable String taxRate, Model model)`
**テンプレート**: `finance/accountsPayable/detail`

#### 2.3.1 処理シーケンス

```
1. transactionMonth を "yyyy-MM" から LocalDate に変換（"-01" を付加）
2. tAccountsPayableSummaryService.getByPK(shopNo, supplierNo, date, new BigDecimal(taxRate))
3. summary が null の場合:
   └ 一覧画面にリダイレクト
4. mSupplierService.getBySupplierNo(supplierNo) で仕入先情報取得
5. Modelに summary, supplier を設定
```

### 2.4 手動検証処理フロー

**エンドポイント**:
`PUT /api/v1/finance/accounts-payable/{shopNo}/{supplierNo}/{transactionMonth}/{taxRate}/verify`
**メソッド**: `TAccountsPayableSummaryService#applyVerification(pk, verifiedAmount, note)`

#### 2.4.1 処理シーケンス

```
1. PKで既存サマリーを取得
2. summary が null でない場合:
   a. verifiedAmount（検証時の税込請求額）をそのまま設定
   b. taxIncludedAmount = verifiedAmount をセット（MF出力スナップショット用）
   c. 税率を百分率に変換: taxRate2 = taxRate / 100 (scale=2, HALF_UP)
   d. 税抜金額算出: taxExcludedAmount = verifiedAmount / (1 + taxRate2)
      └ scale=0, ROUND_DOWN（切り捨て）
   e. 差額計算: difference = verifiedAmount - taxIncludedAmountChange
   f. 一致判定（100円閾値）:
      |difference| <= 100円 → verificationResult = 1（一致）
      |difference| > 100円  → verificationResult = 0（不一致）
   g. verifiedManually = true （SMILE再検証バッチの上書きから保護）
   h. verificationNote = note（DB永続化、VARCHAR 500）
   i. tAccountsPayableSummaryService.save(summary)
3. summary が null の場合 → エラー
```

**手動確定解除**: `DELETE /api/v1/finance/accounts-payable/{pk}/manual-lock`（admin のみ）で
`verifiedManually = false` に戻し、次回 SMILE 再検証バッチでの上書きを許可する。

**振込明細MF一括検証との連携**: `PaymentMfImportService#applyVerification` も同じ
`verifiedAmount` カラムを更新する（税率別の複数行に同一値を書込）。手動検証・Excel一括検証の
どちらも入口は別だが永続化先は同一カラムで整合する。

### 4.4 AccountsPayableSummaryCalculator 集計アルゴリズム

**クラス**: `jp.co.oda32.batch.finance.service.AccountsPayableSummaryCalculator`
**アノテーション**: `@Service`

#### 4.4.1 集計キー（SummaryKey）

**クラス**: `jp.co.oda32.batch.finance.model.SummaryKey`

```java
@AllArgsConstructor
@Getter
@EqualsAndHashCode
public class SummaryKey {
    private final Integer shopNo;
    private final Integer paymentSupplierNo;
    private final String paymentSupplierCode;
    private final BigDecimal taxRate;
}
```

#### 4.4.2 集計アルゴリズム詳細

```
入力: startDate, endDate (集計期間)
処理:
  1. TPurchaseDetail取得:
     tPurchaseDetailService.find(startDate, endDate, Flag.NO)
     ※ del_flg='0'（有効レコード）のみ

  2. Stream処理（除外フィルタ4段）:
     a. filter(除外1): supplier_no != 303（小田光在庫表・手打ち用の特殊仕入先）
        定数: FinanceConstants.EXCLUDED_SUPPLIER_NO
     b. filter(除外2): goods_code NOT IN ('00000021','00000023')
        （第2事業部の月次集約。通常はワーク→本テーブル取込時点で
         WSmilePurchaseOutputFileRepository の NOT IN により既に除外されるが、
         過去データ用の防御フィルタとして残す）
        定数: FinanceConstants.DIVISION2_AGGREGATE_GOODS_CODES
     c. filter(除外3): 論理削除済みマスタ/伝票をスキップ
        - t_purchase.del_flg = '1'
        - m_supplier.del_flg = '1'
        - m_payment_supplier.del_flg = '1'
        スキップ件数は LinkedHashSet に集約し WARN ログで一括出力。
     d. filter(除外4): m_supplier.payment_supplier_no IS NULL
        （親支払先未設定の仕入先は買掛突合キーが作れないため除外）
        スキップ件数は同様に集約してログ出力。

     e. groupingBy:
        キー生成:
          shopNo = FinanceConstants.ACCOUNTS_PAYABLE_SHOP_NO（=1, 固定）
          supplier = detail.getTPurchase().getMSupplier()
          paymentSupplierNo = supplier.getPaymentSupplierNo()
          paymentSupplierCode = supplier.getPaymentSupplier().getPaymentSupplierCode()
          taxRate = detail.getTaxRate()
          → new SummaryKey(shopNo, paymentSupplierNo, paymentSupplierCode, taxRate)

        ※ 重要: 仕入は shop_no=1(第1事業部) と shop_no=2(第2事業部) 両方で発生するが、
          買掛管理は第1事業部が一元管理する運用のため、キーの shopNo は ACCOUNTS_PAYABLE_SHOP_NO
          (=1) で固定化し、shop_no=2 の仕入も shop_no=1 の買掛に合算する。

     f. reducing:
        初期値: TaxAggregationResult(ZERO, ZERO)
        マッピング: detail → TaxAggregationResult(subtotal, ZERO)
                    ※ subtotal = detail.getSubtotal() ?? BigDecimal.ZERO
                    ※ 消費税額は0で初期化（後で一括計算）
        合算: TaxAggregationResult.add()

  3. エンティティ生成:
     FOR EACH (key, aggregation) IN summaryMap:
       IF aggregation.baseAmount > 0 THEN  // 金額ゼロを除外
         taxExcludedAmountChange = aggregation.baseAmount
         taxRate = key.taxRate ?? BigDecimal.TEN  // デフォルト10%
         calculatedTaxAmount = taxExcludedAmountChange * taxRate / 100
                               (scale=0, RoundingMode.HALF_DOWN)
         taxIncludedAmountChange = taxExcludedAmountChange + calculatedTaxAmount

         → TAccountsPayableSummary.builder()
             .shopNo(key.shopNo)                         // 常に 1
             .supplierNo(key.paymentSupplierNo)
             .supplierCode(key.paymentSupplierCode)
             .transactionMonth(endDate)                  // 当月20日
             .taxRate(key.taxRate)
             .taxIncludedAmountChange(taxIncludedAmountChange)
             .taxExcludedAmountChange(taxExcludedAmountChange)
             .build()
       END IF

出力: List<TAccountsPayableSummary>
```

#### 4.4.3 消費税計算の重要ポイント

- 消費税は**明細ごとではなく、支払先・税率単位で一括計算**する
- これにより端数の累積誤差を防止
- 丸めモード: `RoundingMode.HALF_DOWN`（円未満切り捨て相当）
- `taxRate`がnullの場合は10%として処理
- 集計キーは「支払先No + 支払先コード + 税率」の3軸。`m_supplier` は使わず
  `m_payment_supplier`（親支払先）単位で合算する

### 5.4 Step 3: AccountsPayableVerificationTasklet

**クラス**: `jp.co.oda32.batch.finance.AccountsPayableVerificationTasklet`

#### 5.4.1 処理シーケンス

```
1. targetDate → periodEndDate (当月20日) を算出
2. tAccountsPayableSummaryService.findByTransactionMonth(periodEndDate) で全件取得
   ※ verifiedManually=true の行は SmilePaymentVerifier 側でスキップされ、
     taxIncludedAmountChange / verificationResult / paymentDifference は上書きされない
3. SMILE支払情報取得（翌月分）:
   YearMonth.from(periodEndDate).plusMonths(1)
   → tSmilePaymentService.findByYearMonth("yyyyMM")
   ※ 20日締め翌月払いのため、4月20日対象 → 5月の支払情報

4. smilePaymentVerifier.verifyWithSmilePayment(summaries, smilePayments) で照合
   ※ このメソッド内でsummariesのtaxIncludedAmountChangeが調整される
   ※ 手入力保護: verified_manually=true の summary は突合対象から除外され、
     スキップ件数は INFO ログに出力される（運用時の確認用）

5. 調整済みsummariesを一括保存: tAccountsPayableSummaryService.saveAll(summaries)

6. 検証結果をレコードに反映:
   FOR EACH summary IN summaries:
     result = verificationResults.get(summary.supplierCode)
     exist = getByPK(...)  // 最新データ取得
     IF exist != null THEN
       IF result != null THEN
         IF result.adjustedToSmilePayment OR |result.difference| < 5円 THEN
           exist.verificationResult = 1 (一致)
         ELSE
           exist.verificationResult = result.isMatched() ? 1 : 0
         END IF
         exist.paymentDifference = result.difference
       ELSE
         exist.verificationResult = 0 (不一致)  // SMILE支払情報なし
         exist.paymentDifference = null
       END IF
       exist.taxIncludedAmountChange = summary.taxIncludedAmountChange
       exist.taxExcludedAmountChange = summary.taxExcludedAmountChange
       save(exist)
     END IF
```

### 5.5 SMILE支払情報照合ロジック（SmilePaymentVerifier）

**クラス**: `jp.co.oda32.batch.finance.service.SmilePaymentVerifier`

#### 5.5.1 照合アルゴリズム

```
入力: summaries (買掛金サマリーリスト), smilePayments (SMILE支払情報リスト)

1. SMILE支払情報を仕入先コードでグループ化 → smilePaymentMap (supplierCode → paymentAmount合計)
2. 仕入先名情報マップを構築 → supplierNameMap
3. shop_no=2 のマッピング処理:
   MSupplierShopMapping を使って shop_no=2 → shop_no=1 の仕入先コードにマッピング
   → shop2ToShop1SupplierMap
4. 仕入先コードごとに税込合計・税抜合計を算出 → supplierTotalIncTaxMap, supplierTotalExcTaxMap
5. SMILE支払情報が存在しない仕入先を検出:
   → verificationResult = 0, mfExportEnabled = false

6. 仕入先ごとの税率別内訳を計算 → supplierTaxBreakdownMap

7. 各仕入先コードの照合:
   FOR EACH supplierCode IN supplierTotalIncTaxMap.keys:
     a. SMILE支払額の総額取得（マッピングされた仕入先コードも合算）
     b. TaxCalculationHelper.calculateTaxIncludedAmount() で税率別に再計算
     c. 差額計算: truncatedDifference = smilePaymentAmount - recalculatedTaxIncluded
     d. 5円許容差判定:
        |差額| < 5円 AND 差額 != 0 → adjustToSmilePayment = true
     e. 判定結果:
        |差額| < 5円 → 一致
        差額 == 0    → 一致
        |差額| >= 5円 → 不一致
     f. adjustToSmilePayment = true の場合:
        updateAccountsPayableSummaries() でSMILE支払額に合わせる
```

#### 5.5.2 SMILE支払額への調整ロジック

```
メソッド: updateAccountsPayableSummaries(summaries, supplierCode, smilePaymentAmount, originalTaxIncluded)

処理:
  1. 対象仕入先のサマリーを抽出
  2. 税込金額の大きい順にソート
  3. 最大金額のサマリーを選択 → largestSummary
  4. 他のレコードの税込金額合計を算出 → otherSummariesTotal
  5. 最大レコードの調整後金額:
     adjustedAmount = smilePaymentAmount - otherSummariesTotal (scale=0, DOWN)
  6. IF adjustedAmount < 0 THEN
       差額のみ記録し、金額は調整しない
     ELSE
       largestSummary.taxIncludedAmountChange = adjustedAmount
       保存
     END IF
  7. 最終確認: 合計がSMILE支払額と一致するか検証
```

### 5.6 Step 4: AccountsPayableVerificationReportTasklet

**クラス**: `jp.co.oda32.batch.finance.AccountsPayableVerificationReportTasklet`

```

2. 5円未満差額の自動修正:
```

```java
// 集計キー (supplierNo を跨いで shop_no を合算)
```

## 9. 税計算設計

### 9.1 TaxCalculationHelper

**クラス**: `jp.co.oda32.batch.finance.helper.TaxCalculationHelper`
**タイプ**: ユーティリティクラス（staticメソッド）

#### 9.1.1 calculateTaxIncludedAmount

税率ごとの内訳から正確な税込金額を計算する。

```
入力: Map<BigDecimal, TaxBreakdown> taxBreakdownMap
処理:
  totalTaxIncluded = 0
  FOR EACH (taxRate, breakdown) IN taxBreakdownMap:
    taxExcluded = breakdown.getTaxExcludedAmount()
    taxAmount = taxExcluded * taxRate / 100 (scale=0, RoundingMode.DOWN)
    taxIncluded = taxExcluded + taxAmount
    totalTaxIncluded += taxIncluded
  RETURN totalTaxIncluded
エラー:
  taxBreakdownMap が null/空 → IllegalArgumentException
  taxRate が null → IllegalArgumentException
  taxExcluded が null → IllegalArgumentException
```

#### 9.1.2 calculateTaxAmount

税抜価格から消費税額を計算する。

```
入力: baseAmount (税抜金額), taxRate (税率)
処理:
  IF baseAmount == null OR baseAmount == 0 THEN RETURN 0
  IF taxRate == null THEN taxRate = 10 (デフォルト)
  RETURN baseAmount * taxRate / 100 (scale=0, RoundingMode.DOWN)
```

### 9.2 TaxAggregationResult

**クラス**: `jp.co.oda32.batch.finance.model.TaxAggregationResult`

```java
@Getter

public class TaxAggregationResult {
    private final BigDecimal baseAmount;  // 税抜金額合計
    private final BigDecimal taxAmount;   // 消費税額（集計後に一括計算するため0で保持）

    public TaxAggregationResult add(TaxAggregationResult other) {
        return new TaxAggregationResult(
                this.baseAmount.add(other.baseAmount),
                this.taxAmount.add(other.taxAmount)
        );
    }
}
```

**用途**: 買掛金集計時のStream reduce操作で使用。税額は個別明細では計算せず、グループ化後に一括計算する設計。

**用途**: SMILE支払検証時に、仕入先ごとの税率別内訳を保持する。

### 9.4 軽減税率対応

#### 9.4.1 税率体系

| 税率 | 種別 | 仕入用税区分 | 売上用税区分 |
|:----:|------|------------|------------|
| 10% | 標準税率 | `課税仕入 10%` | `課税売上 10%` |
| 8% | 軽減税率 | `課仕 (軽)8%` | `課売 (軽)8%` |
| 0% | 非課税 | `非課税` | `非売` |

#### 9.4.2 税額計算の共通ルール

```
税額 = 税抜金額 * 税率 / 100 (RoundingMode.DOWN)
税込金額 = 税抜金額 + 税額
```

#### 9.4.3 端数処理方針

| 対象 | 丸めモード | 備考 |
|------|-----------|------|
| 消費税額計算（買掛金集計） | `HALF_DOWN` | 五捨六入 |
| 消費税額計算（売掛金集計） | `DOWN` | 切り捨て |
| 消費税額計算（TaxCalculationHelper） | `DOWN` | 切り捨て |
| サービス層save時の金額 | `DOWN`(scale=0) | 整数化 |
| CSV出力時の金額 | `DOWN`(scale=0) | 整数文字列 |
| 税抜金額逆算（税込→税抜） | `DOWN` | 切り捨て |

### 9.5 許容差ルール一覧

| 対象 | 許容差 | 判定 | 処理 |
|------|:------:|------|------|
| 買掛金 vs SMILE支払 | 5円未満 | `|差額| < 5` | 一致扱い、SMILE支払額に合わせる |
| 売掛金 vs 請求書 | 3円以内（設定可） | `|差額| <= tolerance` | 一致扱い、請求書金額に比率調整 |
| 売掛金 1円誤差 | 1円 | `|diff| == 1` | 元の税込金額を採用（警告ログ） |
| 手動検証（買掛金） | 100円以内 | `|差額| <= 100` | 一致とみなす |

### 10.4 出力ファイル名規則

| バッチ | ファイル名パターン |
|--------|-------------------|
| 買掛仕入仕訳（通常） | `accounts_payable_to_purchase_journal_{yyyyMMdd}.csv` |

| 売掛売上仕訳（期間） | `accounts_receivable_to_sales_journal_{fromyyyyMMdd}_{toyyyyMMdd}.csv` |

### 10.5 CSV出力後の確定処理

買掛・売掛とも、CSV出力成功後に以下のコピー処理を実行:

```
FOR EACH summary IN outputSummaries:
  summary.taxIncludedAmount  = summary.taxIncludedAmountChange
  summary.taxExcludedAmount  = summary.taxExcludedAmountChange
  save(summary)
```

これにより作業値（`*Change`）が確定値にコピーされ、出力済みデータとして管理される。

## 付録B: バッチ実行順序

月次処理の推奨実行順序:

```
1. accountsPayableAggregation (買掛金集計)
   パラメータ: targetDate=当月20日
       ↓
2. accountsPayableVerification (買掛金検証 + SMILE照合)
   パラメータ: targetDate=当月20日, smilePaymentFilePath=CSVパス
       ↓
3. purchaseJournalIntegration (買掛仕入仕訳CSV出力)
   パラメータ: targetDate=当月20日, forceExport=false
       ↓
4. accountsReceivableSummary (売掛金集計)
   パラメータ: targetDate=当月末日
       ↓
5. salesJournalIntegration (売掛売上仕訳CSV出力)
   パラメータ: fromDate=前月21日, toDate=当月20日
```
