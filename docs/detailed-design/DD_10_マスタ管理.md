### 1.1 MMakerエンティティ JPA設計

**クラス**: `jp.co.oda32.domain.model.master.MMaker`

```java
@Data
@Builder
@Entity
@AllArgsConstructor
@NoArgsConstructor
@Table(name = "m_maker")
public class MMaker implements IEntity {
    @Id
    @Column(name = "maker_no")
    @SequenceGenerator(
        name = "m_maker_maker_no_seq_gen",
        sequenceName = "m_maker_maker_no_seq",
        allocationSize = 1
    )
    @GeneratedValue(
        strategy = GenerationType.IDENTITY,
        generator = "m_maker_maker_no_seq_gen"
    )
    private Integer makerNo;

    @Column(name = "maker_code")
    private String makerCode;

    @Column(name = "shop_no")
    private Integer shopNo;

    @Column(name = "maker_name")
    private String makerName;

    @Column(name = "del_flg")
    private String delFlg;

    @Column(name = "add_date_time")
    private Timestamp addDateTime;

    @Column(name = "add_user_no")
    private Integer addUserNo;

    @Column(name = "modify_date_time")
    private Timestamp modifyDateTime;

    @Column(name = "modify_user_no")
    private Integer modifyUserNo;

    @Override
    public Integer getShopNo() {
        return null; // 全ショップ共通のため常にnull
    }
}
```

**JPA設計ポイント**:

| 項目 | 設計内容 |
|------|---------|
| 主キー戦略 | `GenerationType.IDENTITY` + PostgreSQLシーケンス `m_maker_maker_no_seq` |
| allocationSize | `1`（PostgreSQLシーケンスのINCREMENT値と一致させる） |
| IEntity実装 | `getShopNo()` は常に `null` を返す。メーカーはショップ横断データであり、`CustomService.validateUpdateByShop()` のショップ権限チェックをスキップする |
| Lombokアノテーション | `@Data`（getter/setter/equals/hashCode/toString自動生成）、`@Builder`（ビルダーパターン）|
| テーブルマッピング | `@Table(name = "m_maker")` |

**フィールドマッピング詳細**:

| Javaフィールド | DBカラム | Java型 | DB型 | NULL可 | 備考 |
|---------------|---------|--------|------|--------|------|
| `makerNo` | `maker_no` | `Integer` | `serial` | NO | PK、シーケンス自動採番 |
| `makerCode` | `maker_code` | `String` | `varchar` | YES | メーカーコード（現在未使用の予備フィールド） |

### 1.4 重複チェックロジック

**新規登録時（MakerCreateController.isExist）**:

```java
private boolean isExist(MakerCreateForm form) {
    List<MMaker> makerList = this.makerService.find(null, form.getMakerName(), Flag.NO);
    return !CollectionUtil.isEmpty(makerList);
}
```

- `find(null, makerName, Flag.NO)` は `makerName` の**部分一致**検索を行う
- 検索結果が1件以上存在する場合、重複とみなす
- 注意: 部分一致であるため、「ABC」を登録しようとした際に既存データに「XABCY」があると誤検出する可能性がある。ただし現行仕様としてはこの動作が意図されている

**更新時（MakerModifyController.isExist）**:

```java
private boolean isExist(MakerModifyForm form) {
    List<MMaker> makerList = this.makerService.find(null, form.getMakerName(), Flag.NO);
    return !CollectionUtil.isEmpty(makerList)
        && makerList.stream().anyMatch(maker -> !maker.getMakerNo().equals(form.getMakerNo()));
}
```

- 新規登録時と同様に部分一致検索を実行
- ただし、**自身（同一makerNo）を除外**して判定する
- 更新前の自身のメーカー名と一致した場合は重複としない

### 1.6 MakerSpecification 検索仕様

**クラス**: `jp.co.oda32.domain.specification.master.MakerSpecification`

**継承**: `CommonSpecification<MMaker>`

| メソッド | 引数 | 生成SQL条件 | null/空時の挙動 |
|---------|------|------------|----------------|
| `makerNoContains(Integer makerNo)` | メーカー番号 | `maker_no = :makerNo` | null の場合は条件なし（null返却） |
| `makerNameContains(String makerName)` | メーカー名 | `maker_name LIKE '%:makerName%'` | null/空文字の場合は条件なし（null返却） |
| `delFlgContains(Flag flag)` | 削除フラグ | `del_flg = :flag` | null の場合は条件なし（CommonSpecificationから継承） |

**条件結合**: `Specification.where(A).and(B).and(C)` により AND 結合される。各メソッドが `null` を返した場合、Spring Data JPA はその条件を無視する。

@Data

    private Integer companyNo;

- `ShopRepository.findById(0)` -> `shop_no = 0 AND shop_no <> 0` となり結果なし
- `@OneToOne` で他エンティティから参照される場合にも `@Where` が適用される

**MShop と MCompany のリレーション**:
- MCompany が `shop_no` を外部キーとして MShop を参照する（逆方向）
- MShop 側から MCompany への直接参照はエンティティレベルでは定義されていない
- `companyNo` フィールドは存在するが、JPA リレーションとしてのマッピングは未定義

## 9. 全マスタ共通パターン

### 9.1 論理削除

**実装箇所**: `CustomService.delete()` メソッド

```java
protected <T extends JpaRepository, U extends IEntity> U delete(T repository, U entity) throws Exception {
    // ログインユーザー情報取得
    entity.setModifyUserNo(loginUser.getUser().getLoginUserNo());
    // 現在日時セット
    entity.setModifyDateTime(DateTimeUtil.getNow());
    // 削除フラグを "1" にセット
    entity.setDelFlg(Flag.YES.getValue());
    // 物理削除ではなく UPDATE で保存
    return (U) repository.save(entity);
}
```

**論理削除の適用パターン**:

| パターン | 適用クラス | 実装方法 |
|---------|-----------|---------|
| Serviceのdeleteメソッド経由 | MMakerService, MWarehouseService 等 | `getOne()` で取得 -> `setDelFlg("1")` -> `CustomService.update()` |
| Controller直接操作 | SupplierShopMappingController | `findById()` で取得 -> `setDelFlg("1")` -> `save()` |

**論理削除データの検索除外**:

| 方式 | 適用箇所 | 実装 |
|------|---------|------|
| Specification | メーカー、倉庫、会社、仕入先、得意先の検索 | `CommonSpecification.delFlgContains(Flag.NO)` |
| `defaultFindAll()` | 各Serviceの `findAll()` | `CustomService.defaultFindAll()` 内で `del_flg='0'` 固定 |
| テンプレート側 | 仕入先店舗マッピング | `th:if="${mapping.delFlg != '1'}"` |
| `@Where` | MShop | Hibernateレベルで `shop_no <> 0` 自動付加 |
| JPQL | MSupplierShopMappingRepository | クエリ内に `AND m.delFlg = '0'` 明示 |

### 9.4 Unicode正規化（NFKC）パターン

メーカー名・倉庫名の登録/更新時に適用される。

```java
String normalizedName = Normalizer.normalize(inputName, Normalizer.Form.NFKC);
```

**NFKC正規化の効果**:
- 全角英数字 -> 半角英数字（例: `Ａ` -> `A`）
- 半角カタカナ -> 全角カタカナ（例: `ｱ` -> `ア`）
- 合字の分解（例: `fi` -> `fi`）
- 重複チェックの精度向上のため、保存前に正規化を行う
